# Test Suite Rebalance Plan

## Context

Phases 1 and 2 of this plan (remove the coverage badge / `scripts/coverage.py` / Makefile target, and trim redundant handler + e2e tests) are already done. What remains is the gap: **the SQL layer has no tests at all.** This document narrows Phase 3 to the parts worth testing and adds a CI step so those tests actually run before deploys.

---

## Why test the repository layer at all?

The fair pushback: `internal/database/{tasks,habits}db/` is generated by sqlc, and testing autogenerated CRUD wrappers is silly — that's testing pgx and sqlc, not our code.

Agreed. So Phase 3 deliberately **does not** test pure pass-through CRUD. It targets the three things in `internal/tasks/repository.go` and `internal/habits/repository.go` that are hand-written, plus the schema constraints those files depend on:

1. **Hand-written error mapping.** `repository.go:350-353` maps pgx error code `23505` to `ErrActiveTimeEntryExists`. If the unique constraint name changes, sqlc upgrades, or the migration drifts, this branch silently stops firing and the handler returns 500 instead of 409. Nothing else catches that.
2. **Hand-written aggregation logic.** `GetProjectChildren` (repository.go:463-611) does bottom-up time accumulation across a project tree in Go: depth-sorted maps, parent rollup, child grouping. None of it is generated. It's the kind of code that works on a 2-level tree and breaks on a 4-level one.
3. **Multi-step orchestration.** `ReplaceTaskDependencies` (repository.go:806-817) issues a delete *then* an upsert. Whether that combination leaves consistent state under real SQL semantics is a property of the pair, not either query alone.

Plus a fourth, slightly different reason:

4. **Schema/migration assertions.** Constraints like `CHECK (task_id != depends_on)` (`006_tasks_dependencies.up.sql`) and the unique active time-entry index (`008_unique_active_time_entry.up.sql`) live in migrations. A repo integration test confirms (a) the migration was applied to the test DB, and (b) the constraint actually fires for the cases our error-mapping expects. That's not "testing sqlc" — it's testing that migrations and Go-side error handling are still in sync.

What Phase 3 explicitly **does not** cover, because it would be silly: trivial `CreateTask` + `GetTask` round-trips, `due_at` clearing (sqlc handles the nullable mapping), simple FK violations on inserts, and any test that just round-trips a struct to the DB and back.

---

## Phase 3 — Add the missing repository tests

Uses the existing test DB plumbing: `docker-compose.test.yaml`, `TEST_DB_URL`, `make test-integration` (`-run Integration` filter). No new infrastructure. Tests `t.Skip` when `TEST_DB_URL` is unset so `go test ./...` without the integration env stays fast.

### 3.1 `internal/testutil/db.go` — small helper

Why: every repo test needs the same three things (open pool, truncate between runs, skip cleanly when env var missing). Inlining triples the line count. Keep it deliberately small — no fixtures DSL.

```go
// internal/testutil/db.go
package testutil

import (
	"context"
	"os"
	"testing"

	"github.com/jackc/pgx/v5/pgxpool"
)

func NewPool(t *testing.T) *pgxpool.Pool {
	t.Helper()
	url := os.Getenv("TEST_DB_URL")
	if url == "" {
		t.Skip("TEST_DB_URL not set; skipping integration test")
	}
	pool, err := pgxpool.New(context.Background(), url)
	if err != nil {
		t.Fatalf("connect test db: %v", err)
	}
	t.Cleanup(pool.Close)
	return pool
}

func Truncate(t *testing.T, pool *pgxpool.Pool, tables ...string) {
	t.Helper()
	ctx := context.Background()
	for _, table := range tables {
		if _, err := pool.Exec(ctx, "TRUNCATE TABLE "+table+" RESTART IDENTITY CASCADE"); err != nil {
			t.Fatalf("truncate %s: %v", table, err)
		}
	}
}
```

### 3.2 `internal/tasks/repository_integration_test.go` — four tests

Each test maps to one of the four reasons above. Trivial CRUD round-trips are deliberately omitted.

```go
// internal/tasks/repository_integration_test.go
package tasks_test

import (
	"context"
	"errors"
	"testing"
	"time"

	"gv-api/internal/database/tasksdb"
	"gv-api/internal/tasks"
	"gv-api/internal/testutil"

	"github.com/stretchr/testify/require"
)

func newRepo(t *testing.T) *tasks.PostgresRepository {
	pool := testutil.NewPool(t)
	testutil.Truncate(t, pool,
		"task_dependencies", "time_entries", "todos", "tasks", "projects",
	)
	return tasks.NewRepository(tasksdb.New(pool))
}

// (1) Hand-written error mapping + (4) constraint assertion.
// If the unique-active-time-entry index goes missing or the pg error code
// changes, this test fails before the bug reaches production.
func TestIntegration_ActiveTimeEntryUnique(t *testing.T) {
	ctx := context.Background()
	repo := newRepo(t)

	task, err := repo.CreateTask(ctx, nil, "t", nil, nil, "single", nil, 4)
	require.NoError(t, err)

	_, err = repo.CreateTimeEntry(ctx, task.ID, time.Now(), nil, nil)
	require.NoError(t, err)

	_, err = repo.CreateTimeEntry(ctx, task.ID, time.Now(), nil, nil)
	require.True(t, errors.Is(err, tasks.ErrActiveTimeEntryExists),
		"second open time entry should return ErrActiveTimeEntryExists, got %v", err)
}

// (2) Hand-written aggregation logic. Bottom-up time accumulation across a
// 3-level tree exercises the depth-sort + parent-rollup paths in
// GetProjectChildren that no smaller tree would cover.
func TestIntegration_FinishProjectTreeCascades(t *testing.T) {
	ctx := context.Background()
	repo := newRepo(t)

	root, err := repo.CreateProject(ctx, "root", nil, nil, nil)
	require.NoError(t, err)
	mid, err := repo.CreateProject(ctx, "mid", nil, nil, &root.ID)
	require.NoError(t, err)
	leaf, err := repo.CreateProject(ctx, "leaf", nil, nil, &mid.ID)
	require.NoError(t, err)

	leafTask, err := repo.CreateTask(ctx, &leaf.ID, "t", nil, nil, "single", nil, 4)
	require.NoError(t, err)

	require.NoError(t, repo.FinishDescendantProjects(ctx, root.ID))
	require.NoError(t, repo.FinishTasksByProjectTree(ctx, root.ID))

	got, err := repo.GetTask(ctx, leafTask.ID)
	require.NoError(t, err)
	require.NotNil(t, got.FinishedAt, "leaf task should be finished by cascade")

	gotProj, err := repo.GetProject(ctx, leaf.ID)
	require.NoError(t, err)
	require.NotNil(t, gotProj.FinishedAt, "leaf project should be finished by cascade")
}

// (3) Multi-step orchestration. Replace = delete + upsert; verifies the pair
// leaves consistent state across grow, shrink, and clear. A task can depend
// on multiple tasks simultaneously.
func TestIntegration_ReplaceTaskDependencies(t *testing.T) {
	ctx := context.Background()
	repo := NewRepo(t)

	mk := func(name string) int32 {
		task, err := repo.CreateTask(ctx, nil, name, nil, nil, "standard", nil, 4)
		require.NoError(t, err)
		return task.ID
	}
	a, b, c, d := mk("a"), mk("b"), mk("c"), mk("d")

	depIDs := func(t *testing.T, id int32) []int32 {
		t.Helper()
		deps, _, _, err := repo.GetTaskDependencies(ctx, id)
		require.NoError(t, err)
		out := make([]int32, len(deps))
		for i, dep := range deps {
			out[i] = dep.ID
		}
		return out
	}

	require.NoError(t, repo.ReplaceTaskDependencies(ctx, a, []int32{b, c}))
	require.ElementsMatch(t, []int32{b, c}, depIDs(t, a))

	require.NoError(t, repo.ReplaceTaskDependencies(ctx, a, []int32{b, c, d}))
	require.ElementsMatch(t, []int32{b, c, d}, depIDs(t, a))

	require.NoError(t, repo.ReplaceTaskDependencies(ctx, a, []int32{d}))
	require.ElementsMatch(t, []int32{d}, depIDs(t, a),
		"replace should remove old deps not in the new set")

	require.NoError(t, repo.ReplaceTaskDependencies(ctx, a, []int32{}))
	require.Empty(t, depIDs(t, a))
}

// Circular dependencies must be rejected. The schema's
// CHECK (task_id != depends_on) only blocks self-deps, so cycle detection
// belongs in the repository or service layer. This test asserts the wanted
// behavior — it will fail (red) until cycle detection is implemented.
func TestIntegration_CircularTaskDependenciesRejected(t *testing.T) {
	ctx := context.Background()
	repo := NewRepo(t)

	mk := func(name string) int32 {
		task, err := repo.CreateTask(ctx, nil, name, nil, nil, "standard", nil, 4)
		require.NoError(t, err)
		return task.ID
	}
	a, b := mk("a"), mk("b")

	require.NoError(t, repo.ReplaceTaskDependencies(ctx, a, []int32{b}))
	require.Error(t, repo.ReplaceTaskDependencies(ctx, b, []int32{a}),
		"creating a cycle (A→B, B→A) must be rejected")
}
```

> Note: confirm exact field/method shapes against the current repo code when implementing — names like `CreateTask`'s argument order are taken from `repository.go` as of writing.

### 3.3 `internal/habits/repository_integration_test.go` — one test

The only non-trivial habits SQL is the period-aware grouping. Everything else is sqlc CRUD.

```go
// internal/habits/repository_integration_test.go
package habits_test

import (
	"context"
	"testing"
	"time"

	"gv-api/internal/database/habitsdb"
	"gv-api/internal/habits"
	"gv-api/internal/testutil"

	"github.com/stretchr/testify/require"
)

// `period_test.go` already covers groupLogsByPeriod against synthetic data.
// This test confirms the same logic still works when the rows come from real
// SQL with real DATE columns straddling an ISO-week boundary.
func TestIntegration_HabitLogs_AcrossPeriodBoundary(t *testing.T) {
	ctx := context.Background()
	pool := testutil.NewPool(t)
	testutil.Truncate(t, pool, "habit_logs", "habits")
	repo := habits.NewRepository(habitsdb.New(pool))

	h, err := repo.CreateHabit(ctx, "weekly", nil, "weekly", nil, nil, true)
	require.NoError(t, err)

	mustLog := func(iso string, val float64) {
		d, _ := time.Parse("2006-01-02", iso)
		_, err := repo.LogValue(ctx, h.ID, d, val)
		require.NoError(t, err)
	}
	// Sun 2026-04-26 and Mon 2026-04-27 — different ISO weeks.
	mustLog("2026-04-26", 1.0)
	mustLog("2026-04-27", 2.0)

	groups, err := repo.GetLogsGroupedByPeriod(ctx, h.ID)
	require.NoError(t, err)
	require.Len(t, groups, 2, "logs straddling Sun/Mon should fall into two ISO weeks")
}
```

> Note: `CreateHabit`, `LogValue`, `GetLogsGroupedByPeriod` are placeholders — match against the actual `habits.Repository` interface when implementing. If the equivalent method doesn't exist yet, this test pins down what it needs to do.

## Phase 4 — Wire integration tests into the deploy pipeline

This is what makes the constraint-assertion argument from Phase 3 *actually* useful. Today `.gitea/workflows/deploy.yml` does this on every push to `main`:

```yaml
- name: Deploy
  run: |
    git config --global --add safe.directory /home/ocp/docker/gv/gv-api
    cd /home/ocp/docker/gv/gv-api
    git pull
    docker compose up --build --wait -d
```

No tests. The deploy pulls and rebuilds. Which means: a migration that no longer matches the Go-side error mapping ships to prod, the repo integration test from Phase 3 never runs, and the first time anyone notices is a 500 in production.

**The fix**: run `make test` on the deploy host before `docker compose up --build`. If tests fail, the deploy aborts and the previous container keeps running.

### 4.1 `.gitea/workflows/deploy.yml` — run tests before deploying

```yaml
name: Deploy

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: self-hosted
    steps:
      - name: Pull latest
        run: |
          git config --global --add safe.directory /home/ocp/docker/gv/gv-api
          cd /home/ocp/docker/gv/gv-api
          git pull

      - name: Run tests
        run: |
          cd /home/ocp/docker/gv/gv-api
          make test

      - name: Deploy
        run: |
          cd /home/ocp/docker/gv/gv-api
          docker compose up --build --wait -d
```

Why split into three steps: if the test step fails, Gitea Actions stops the job before `docker compose up` runs, so the running container is untouched. With the current single-step script, a `set -e` failure in the middle still leaves the deploy half-done.

### 4.2 Things to verify before merging this pipeline change

- `make test` requires `go`, `docker`, `docker compose`, and a writable `.env` on the runner. The runner is already self-hosted and runs `docker compose`, so `go` is the only likely missing piece — install it on the host or run tests inside a Go container as part of the step.
- `make test` brings up the `db` service, creates a test DB, runs unit + integration + e2e, then tears the test DB down. It does *not* touch the running production container (`make test-api-setup` rebuilds `gv-api` against the test DB; `make test-db-cleanup` restores it). Confirm this on a staging host before pointing it at prod.
- The deploy host would now need to be able to run the full test suite without affecting the running app, even briefly. If that's a concern, the alternative is to run tests on the *runner* (in a clean checkout, against a throwaway DB) and only the deploy step on the prod host. That's a bigger change and probably the right long-term shape, but out of scope for this pass.

### 4.3 Why this pairs with Phase 3 specifically

The argument for repo integration tests rests partly on "they catch migration drift before deploy." That argument is empty if the tests don't run before deploy. Either both Phase 3 and Phase 4 land, or neither is worth doing — repo tests that only run on a developer's laptop catch drift only by coincidence.

---

## Out of scope

- Refactoring service or repository code itself.
- Introducing testcontainers — the existing docker-compose test DB works.
- Moving CI off the deploy host onto a separate runner. Worth doing eventually; not part of this plan.
